### 2. 技术上下文
- **基础框架**: Next.js (Chat Simulator Pro)
- **核心组件**: [DiscordChatSimulator](https://github.com/lukepolson/DiscordChatSimulator)
- **样式方案**: Tailwind CSS
- **核心逻辑**: 需将静态渲染转为动态 State 管理，利用 `URL.createObjectURL` 处理临时图片，导出时优先使用“安全内联样式快照 + PNG 导出”方案，避免被全局主题中的 `oklab/oklch/lab` 颜色函数干扰。
- **补充指令**: 请注意，本项目不使用任何数据库。所有数据管理需在前端通过 React state 完成。对于头像上传功能，请使用 `URL.createObjectURL` 实现本地预览；如果需要持久化，请优先考虑 `localStorage`。头像应视为“身份级配置”而非单条消息的孤立字段：当某个发言人头像被修改时，默认同步所有同名历史消息和对应用户配置，保持整段对话视觉一致。

### 2.1 本次升级实际使用的技术栈
- **React Hooks**: 使用 `useState`、`useEffect`、`useRef` 管理编辑态、播放态、导出态、身份配置和临时图片对象链接。
- **Zustand Store**: 统一管理消息、消息分组、用户缓存、频道配置，以及 `add / update / delete / startNewChat / updateAuthorAvatar` 等核心动作。
- **TypeScript 类型建模**: 为 `ChatMessage`、`ChatUser`、`ChatChannel`、`MessageGroup` 等结构补充 `authorName`、`avatarUrl`、频道信息和导出所需字段，保证多平台复用时类型稳定。
- **Tailwind CSS + Scoped Skin CSS**: 负责整体布局、Discord 深色主题、输入框高对比修复、设置卡片、欢迎区、编辑框和 hover 操作栏样式。
- **浏览器原生文件能力**: 使用 `<input type="file" />` + `URL.createObjectURL` 处理头像、群头像、背景图的本地预览，不依赖上传服务器。
- **原生导出链路**: 使用安全快照与 PNG 导出能力生成图片，避免第三方截图库直接解析宿主项目全局主题导致的 `oklab/oklch/lab/color-mix` 报错。
- **导出资源净化**: 导出前应把头像、群头像、背景图等图片资源尽量内联成 `data URL`，确保导出用的 `canvas` 保持 `origin-clean`，避免 `toBlob`、`toDataURL` 因跨源污染报错。
- **设备分流交付**: 桌面端默认触发文件下载；手机与平板端优先使用 `navigator.share` 调起系统分享面板，让用户保存到相册或文件。
- **前端异步播放控制**: 使用 `async/await + setTimeout` 实现逐条播放、停止播放、重新播放，以及“同一人连续发言约 1 秒一条、切换发言人时 2-3 秒随机停顿”的节奏控制。
- **导出参数控制**: 为图片导出提供“比例预设”和“清晰度预设”，方便在不同投放场景下复用。
- **平台切换能力**: 工具内使用统一状态管理维护 `activeSkin`，并通过平台选择器切换不同聊天平台的结构和视觉样式。

### 3. 分阶段执行策略（请按此顺序引导我执行）

#### 第一阶段：数据结构重构与基础 CRUD
- **目标**: 让聊天记录“活”起来。
- **任务**:

    3. 实现“新建聊天”按钮，点击后清空当前状态并初始化默认欢迎语。

#### 第二阶段：身份系统与临时素材
- **目标**: 允许实时定义发送者。
- **任务**: 
    1. 在输入框左侧或上方建立“身份配置区”。
    2. 实现临时头像上传：用户上传图片后，生成 `blob` 链接作为 `src`，不上传服务器。
    3. 支持输入框实时修改“当前发言人名称”。
    4. 身份配置区必须提供公共的“发言气泡位置 / Message side”设置项，允许用户明确指定下一条消息渲染在左侧还是右侧；该能力需要适用于所有平台。
    5. 确保消息对象中包含 `authorName` 和 `avatarUrl` 字段。
    6. 在“编辑消息”弹出的编辑框内，也要支持直接修改该发言人的头像。
    7. 当编辑框或身份配置区修改了某个发言人的头像后，默认同步所有同名消息与同名用户配置，而不是只修改当前这一条消息。
    8. 默认示例数据应带真实可用头像，不允许引用不存在的演示图片路径。
    9. 编辑消息时还必须支持修改该条消息的发送时间，并立即反映到当前平台对应的时间显示中。
    10. 对于支持附件的平台，附件按钮和相机按钮必须真正可用：用户选择本地文件后，应以纯前端临时资源形式展示在聊天中，不上传服务器、不写入后端。

#### 第三阶段：界面高度仿真与背景定制
- **目标**: 1:1 还原 Discord 体验。
- **任务**: 
    1. 增加“群组设置”面板，允许修改：群名称、群头像、频道简介。
    2. 增加“环境设置”，支持上传聊天背景图或修改背景色。
    3. 优化 CSS，确保字体、间距、颜色（如 `#313338` 背景色）与 Discord 官方一致。
    4. 频道设置修改后，顶部 Header、聊天欢迎区与“新建聊天”默认欢迎语都应实时同步，不允许出现配置区和渲染区状态割裂。
    5. 背景设置应直接作用于消息滚动区域，并提供实时预览；背景图应与半透明遮罩叠加，避免影响消息可读性。
    6. 设置面板本身也需要保持 Discord 风格：深色卡片、细边框、轻微阴影和一致的输入框配色。
    7. 输入框、身份输入框、消息编辑框都必须锁定高对比深色底和浅色文字；即使宿主项目有全局 `input/textarea` 样式覆盖，也不能出现白底浅字或灰字看不清的问题。
    8. 设置区默认应折叠隐藏，并放置在聊天预览框外部下方；用户展开后再进行群组、环境、导出和身份配置。
    9. 聊天预览框默认高度应接近整屏，以保证打开页面就能看到足够多的消息；当消息总高度超过阈值后，只允许消息区内部滚动。

#### 第四阶段：动态播放与图片导出
- **目标**: 增加内容产出的交互感。
- **任务**: 
    1. **播放功能**: 实现 `playChat` 函数，通过 `setTimeout` 或 `async/await` 按顺序逐条渲染消息，模拟真人对话过程。
    2. **导出功能**: 捕获 `.discord-chat-container` 容器，先为导出节点内联安全的计算样式，再生成并下载高质量 PNG 图片；不要依赖会直接解析全局样式表的方案，以免被 `oklab/oklch/lab` 颜色函数击穿。
    3. 播放期间应支持“停止播放”而不是只能等到结束；开始播放时应从第一条消息重新开始逐条出现。
    4. 播放节奏应更接近真人：同一人连续多条消息时按约 `1` 秒一条出现；切换到另一位发言人时，插入 `2-3` 秒随机停顿。
    5. 导出时应自动隐藏配置面板、身份编辑区等控制 UI，仅保留聊天预览本身，避免把后台控制项一起导出进截图。
    6. 导出的文件名应基于当前频道名生成，默认输出 PNG；导出前应确保编辑态已退出，避免把编辑框误导出。
    7. 如果宿主项目存在全局主题变量、`oklab/oklch/lab/color-mix` 等高级颜色函数，导出功能必须使用安全快照方案，不能把颜色解析风险留给运行时。
    8. 如果导出涉及头像、背景图或上传素材，必须保证导出画布是 `origin-clean`；必要时先把资源转成 `data URL` 后再渲染到导出画布。
    9. 导出设置应允许用户选择比例和清晰度，用于适配不同平台的图片投放场景。
    10. 每新增一个聊天平台，都需要补齐平台切换、头部结构、消息气泡、输入区、设置区和导出预览的对应样式，不允许只换颜色不换结构。

- 工具栏与平台选择器需常驻聊天框外部顶部，并兼容不同分辨率下的换行与布局。
- WhatsApp 附件上传已纳入当前实现：附件/相机按钮必须可用，所选文件仅作为前端临时消息附件展示，不上传服务器。
- 设置区显隐与布局已纳入统一标准：默认隐藏、外置到聊天框下方、通过顶部工具栏控制。
- 聊天预览高度与滚动策略已纳入统一标准：默认接近整屏，消息过多时只在聊天框内部滚动。
- WhatsApp 日期与时间规则已纳入统一标准：日期分隔条居中显示在聊天流中，消息时间保持在消息内容后方。
