# 工具集成引擎第一阶段：技术实施方案

根据《ToolingEngineArchitecture.md》的要求，本文档为第一阶段“功能规划与架构预研”的技术实施方案。在获得本方案的审核通过指令前，不会输出任何具体业务代码。

## 1. 多态架构目录树

系统分为 `themes` (主题引擎) 和 `tools` (工具功能) 两大部分，为了实现高内聚和热插拔能力，所有的基础 UI 抽象和皮肤规范均在工具内闭环。

```text
src/
├── themes/
│   └── [theme-name]/             # 主题容器，负责全局布局和路由解析
│       ├── layout.tsx            # 提供 Header、Stage、Sidebar 等插槽
│       ├── page.tsx
│       ├── globals.css           # 定义全局统一预置色带及 CSS Variables (如 --primary-color)
│       └── components/           # 主题特有的基础 UI 及框架组件
└── tools/
    └── [tool-name]/              # 独立可插拔功能包 (例如: DiscordChatSimulator)
        ├── manifest.json         # 工具配置文件 (SEO/GEO/路由模式/皮肤装载项)
        ├── index.tsx             # 工具入口，根据 manifest.type 决定导出普通组件或页面
        ├── store/
        │   └── index.ts          # 工具独立的本地 Store (Zustand/Context)，禁止全局共享与强耦合
        ├── components/           # 提取出的无样式头部 UI(Headless UI) 原子组件库
        │   ├── ChatContainer.tsx
        │   ├── MessageBubble.tsx
        │   ├── SenderInfo.tsx
        │   └── InputBar.tsx
        ├── skins/                # 动态皮肤目录，放置只与视觉相关的覆写变量或模块化类
        │   ├── discord.css       # Discord 风格特定的 CSS 变量覆盖与专属微调样式
        │   ├── whatsapp.css      # WhatsApp 风格特定的 CSS 变量覆盖与专属微调样式
        │   └── telegram.css      # Telegram 风格特定的 CSS 变量覆盖与专属微调样式
        └── utils/
            └── polymorphicEngine.ts # 运行时渲染和皮肤挂载及协议处理辅助函数
```

- **核心原则**: `tools/[tool-name]/skins/` 集中管理当前工具的多套视觉表现。使用加载对应 stylesheet 或者不同的 CSS Variable 挂载作用域类名，实现不同平台 UI 的无缝秒切换。

## 2. 抽象数据协议 (Data Schema)

为了统一 Discord、WhatsApp、Telegram 等平台的消息模型结构，本工具建立以下标准联合结构：

```typescript
// src/tools/[tool-name]/types/chat.ts
export type PlatformType = 'discord' | 'whatsapp' | 'telegram' | 'default';

export interface User {
  id: string;
  name: string;
  avatar: string; // 头像链接
  status?: 'online' | 'offline' | 'idle' | 'dnd'; // 平台活动状态 
  role?: string;  // 角色标识 (如 Admin/Bot/Guest)
  badges?: string[]; // 图标徽章集合 (兼容 Discord 特性)
}

export interface Attachment {
  id: string;
  type: 'image' | 'video' | 'file' | 'audio';
  url: string;
  name: string;
  size?: number; // 以 byte 为单位
  thumbnail?: string; // 视频或图片的专用低分预览图
}

export interface Reaction {
  emoji: string;
  count: number;
  users: string[]; // 点赞用户的 ID 集合
}

export interface ChatMessage {
  id: string;
  sender: User;
  content: string; // 消息文本主体，通常允许简单 Markdown 解析或纯文本
  timestamp: string; // ISO 8601 标准时间格式
  isSystemMessage?: boolean; // 区分用户自然语意与系统交互通知
  attachments?: Attachment[]; // 媒体及外部附件挂载
  reactions?: Reaction[]; // 消息表情表态/点赞交互
  replyTo?: string; // 引用及回复消息类型的关联 ID
  metadata?: Record<string, any>; // 平台专属特征扩充字段 (例如保存 WhatsApp 的“已读/未读”标记)
}
```

## 3. 皮肤切换架构 (Skinning System)

采用 **"Tailwind + CSS Variables动态化 + 数据驱动注册"** 混合机制，确保逻辑表现分离。

1. **结构表现解耦**：基础原子组件 (`MessageBubble`等) 的内部布局只编写结构性（Flexbox/Grid/Spacing）和无特定状态背景色的 Tailwind CSS。
2. **主题插槽注入**：在渲染前，根据 `manifest.json` 中定义或用户传入的 `skin` 向 DOM 根节点注入皮肤层范围类，如 `<div className="skin-theme-discord">`。
3. **样式变量重载**：在不同的皮肤样式文件中定义同组变量族群，由特定类限制作用域：

```css
/* src/tools/[tool-name]/skins/discord.css */
.skin-theme-discord {
  --ds-chat-bg: #313338;
  --ds-bubble-bg: transparent;
  --ds-text-primary: #dbdee1;
  --ds-text-secondary: #949ba4;
  --ds-font-family: 'gg sans', 'Helvetica Neue', Helvetica, Arial, sans-serif;
  --ds-border-radius: 4px;
  --ds-accent-color: var(--primary-color); /* 继承 theme 全局色彩做主调 */
}

/* src/tools/[tool-name]/skins/whatsapp.css */
.skin-theme-whatsapp {
  --ds-chat-bg: #e5ddd5;
  --ds-bubble-bg: #d9fdd3;
  --ds-text-primary: #111b21;
  --ds-text-secondary: #667781;
  --ds-font-family: 'Segoe UI', 'Helvetica Neue', sans-serif;
  --ds-border-radius: 8px;
  --ds-accent-color: #25d366;
}
```

4. **原子化调用桥接**：所有工具内组件强制使用自定义变量或 Tailwind Arbitrary Properties (`bg-[var(--ds-chat-bg)]`) 处理所有边框和色彩参数。

## 4. 样式隔离方案

严格推行 **作用域限定 (Scoped Isolation)** 去防止全局干扰，这对于插件机制至关重要。

1. **唯一前缀控制 (Prefix Proxy)**:
   在此工具的所有手写类上应用特有的命名空间（例如 `ds-` 或当前 tool 的命名首字母缩写。如果使用 `Tailwind Prefix`，需将其仅限定在此 `tools/` 根目录配置内（通过独立 postcss/tailwind 配置或通过统一变量命名如 `--ds-`）约束影响。

2. **强约束顶层作用域**:
   禁止给 `body` 或 `html` 注入任何重置样式。所有的基准样式重置只允许在每个 Tool 包装组件的顶层 ID/类 下派发，实现沙箱效果：
   ```css
   .tool-root-discord-simulator * {
       /* 重置仅限内部流 */
       box-sizing: border-box;
   }
   ```
3. **模块化补充 (CSS Modules)**:
   对第三方外部不可控样式的覆写，以及部分无法被纯 Tailwind 表达的复杂自绘动效动画，必须使用 `.module.css` 的后缀强制散列化命名，确保打包后与其余工具彻底互不干扰。

## 6. UI 适配逻辑

根据预留机制准备后续的专业级 UE / UI 定制工作对接：

1. **核心骨架与槽位抽象 (Skeleton & Slots)**
   主题 `theme` 容器内部使用 `<ThemeLayout>` 并只提供带约定宽度的框架插槽，如 `<SidebarArea />`和 `<MainStage />`。
2. **事件与属性渗透 (Props Flow)**
   工具在以 `type: "component"` 加载时，会暴露标准 React Context 下放所需的组件槽（如提供用于挂载其自身功能配置面板到大主题左侧控制台的 Slot ReactNode）。
3. **视觉驱动还原迭代**
   当收到正式 UE/UI 设计文件后：
   - 全局版式微调：仅修改由 `themes` 传导的内容约束布局。
   - 工具 UI 定制：通过建立具体的 `skins/[name].css` 映射相应的阴影、渐变、边距和色块参数。
   - 实现组件级别的 “设计稿参数 => CSS Variables” 的 1-to-1 直接翻译。
